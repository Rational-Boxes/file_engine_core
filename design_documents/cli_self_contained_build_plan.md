# Self-Contained CLI Build — Plan

**Status:** Draft
**Scope:** Producing a `fileengine_cli` binary with minimal-to-zero runtime
dependencies, suitable for dropping onto an arbitrary Linux host to talk to a
remote FileEngine server.
**Relates to:** `cli/CMakeLists.txt` (`BUILD_STATIC_CLI` target),
`fileengine-core.spec` (`fileengine-cli-static` subpackage).

---

## 1. Goal

Ship a client binary that runs on a machine where *nothing else FileEngine is
installed* and where we cannot assume a matching C++ runtime or a particular
set of `-devel` packages. The user-facing promise is: **copy one file, run it,
talk to the server.**

### Success criteria

| Tier | Definition | `ldd` result |
|------|------------|--------------|
| T0 (today) | Static libstdc++/libgcc, no `libfileengine_core.so` | ~74 dynamic libs (gRPC graph) |
| T1 | All third-party libs static; only glibc + loader dynamic | ~3–4 libs (`libc`, `libm`, `libpthread`/`libdl`, loader) |
| T2 | Fully static, no loader | `not a dynamic executable` |

T1 is the realistic, robust target. T2 is achievable only on a musl toolchain
(see §6, Option D) and is a stretch goal.

---

## 2. Current state (T0)

`-DBUILD_STATIC_CLI=ON` produces `fileengine_cli_static`, which already:

- **drops `libfileengine_core.so`** — the CLI only uses the header-only logger
  (`cli/include/logger.h`) plus the static `proto_lib`, so the shared core
  library was a needless link;
- **statically links libstdc++ / libgcc** (`-static-libstdc++ -static-libgcc`),
  so it no longer needs a matching `libstdc++.so.6` / CXXABI on the target;
- uses `-Wl,--as-needed` to drop unreferenced libraries.

It still dynamically links the entire gRPC dependency graph because **Fedora
ships no static archives** for any of it (confirmed: `protobuf-static`,
`grpc-static`, `openssl-static`, `libpq-static`, `abseil-cpp-static`,
`re2-static`, `c-ares-static` are all absent from the repos; only
`glibc-static`, `libstdc++-static`, and `zlib-ng-compat-static` exist).

The dependency floor is therefore set by **gRPC being dynamic**: statically
linking our own libstdc++ is partially undone because the dynamic `libgrpc++`,
`libprotobuf`, and the ~50 `libabsl_*` libraries each re-introduce a
`libstdc++.so.6` NEEDED entry transitively.

**Conclusion:** moving below T0 requires building the heavy stack from source.
System packages cannot get us there.

## 3. Dependency inventory

Versions currently linked (from the running binary):

| Library | Current ver | Static build method | Gotchas |
|---|---|---|---|
| gRPC C++ | `libgrpc++.so.1.48` / `libgrpc.so.26` | CMake (`gRPC_BUILD_*`) | Pulls everything below; needs matching protobuf/abseil |
| protobuf | `libprotobuf.so.30` | CMake (`protobuf_BUILD_SHARED_LIBS=OFF`) | Must match the `protoc` used to generate `fileservice.pb.*` |
| Abseil | `2508.x` (~50 libs) | CMake | Version-locked to gRPC; ABI churn is severe — build from the gRPC-pinned tag |
| OpenSSL | `libssl.so.3` / `libcrypto.so.3` | `./Configure no-shared` | `no-engine`/`no-dso` to avoid runtime `dlopen`; FIPS off |
| c-ares | `libcares.so.2` | CMake (`CARES_STATIC=ON`) | gRPC async DNS resolver |
| re2 | `libre2.so.11` | CMake | Depends on abseil |
| zlib | `libz.so.1` | system `zlib-ng-compat-static` or source | Easiest dep |
| libpq | `libpq.so.5` | build from postgresql source `--without-shared` | Drags in OpenSSL, optionally GSS/LDAP — disable those |
| ICU | `libicuuc.so.77` / `libicudata.so.77` | source | Surprise dep via protobuf/abseil; `libicudata` is ~30 MB — see §9 |

**Note on the CLI's actual needs:** the client never touches PostgreSQL. The
`libpq` dependency is inherited only because `proto_lib`/the link list pulls it
in. A worthwhile early task is to confirm the CLI has **zero** libpq symbol
references (`nm -DC ... | grep PQ`) and, if so, drop `LIBPQ_LIBRARIES` from the
static target entirely — removing one of the harder static deps for free.

## 4. Constraints & hazards

1. **glibc static linking is fragile.** `-static` against glibc makes
   `getaddrinfo`/NSS use `dlopen` at runtime anyway, emitting warnings and
   risking name-resolution failures — exactly what gRPC's resolver needs. This
   is why T1 (glibc dynamic, everything else static) is the safe target and T2
   requires musl.
2. **OpenSSL engines/providers.** A static OpenSSL that still `dlopen`s
   providers is not self-contained. Configure with `no-shared no-dso no-engine`
   and (OpenSSL 3) link the default+legacy providers statically.
3. **Abseil ABI lock-step.** gRPC, protobuf, and Abseil must come from one
   coherent set of pinned versions; mixing distro protobuf headers with a
   source-built Abseil produces link-time chaos. Build all three from the
   versions gRPC pins.
4. **`protoc` consistency.** `fileservice.pb.*` must be generated by the *same*
   protobuf the binary links, or descriptor/ABI mismatches appear. The
   superbuild must expose its own `protoc`.
5. **Link order for static archives.** Static libs are order-sensitive; wrap the
   group in `-Wl,--start-group ... -Wl,--end-group` (or rely on the package
   manager's generated link order).
6. **Licensing.** OpenSSL (Apache-2.0 in 3.x), gRPC/protobuf/Abseil (Apache-2.0),
   re2 (BSD), c-ares (MIT), zlib (zlib), libpq (PostgreSQL). All permit static
   linking and redistribution; record attributions in the distributed artifact.
   No copyleft blockers.

---

## 5. Why not "just install the SRPMs"

Installing the dependency SRPMs only unpacks upstream source + spec into
`~/rpmbuild`; it installs no libraries. Getting static archives that way means
*rebuilding* each binary RPM with static enabled — but the Fedora specs do not
build static libs (hence the missing `-static` subpackages), so each spec would
need editing (`--enable-static`, `.a` into `%files`) and rebuilding in
dependency order (abseil → protobuf → gRPC, plus openssl, c-ares, re2, zlib).
That is a from-source static build wearing an RPM costume — strictly more
friction than the options below, with no upside. **Rejected.**

---

## 6. Options

### Option A — vcpkg static triplet (recommended)

Use `vcpkg` with the `x64-linux` triplet (static libs, glibc dynamic) to build
`grpc`, `protobuf`, `openssl`, and friends, then point the CLI's CMake at the
vcpkg toolchain.

- **Pros:** vcpkg already encodes the gRPC↔protobuf↔abseil version graph and
  static build flags; one `vcpkg install grpc` resolves the whole tree;
  reproducible via a manifest (`vcpkg.json`) + baseline commit; integrates with
  CMake via a single `-DCMAKE_TOOLCHAIN_FILE`.
- **Cons:** first build is long (~30–60 min) and disk-heavy (~a few GB); adds a
  submodule/clone to the build environment.
- **Effort:** Low-Medium.

### Option B — CMake `ExternalProject` superbuild

Hand-write a top-level superbuild that fetches/builds each dependency static at
pinned tags, then builds the CLI against them.

- **Pros:** no external tool; full control of flags; lives entirely in-repo.
- **Cons:** we re-implement what vcpkg/Conan already maintain (the abseil/gRPC
  version matrix, OpenSSL `Configure` invocation, install layout); highest
  maintenance burden.
- **Effort:** High.

### Option C — Conan 2

Declare deps in a `conanfile`, set `*:shared=False`, build/link via the Conan
CMake toolchain.

- **Pros:** like vcpkg, encodes the graph; good binary caching; `protoc`
  packaged with protobuf.
- **Cons:** another toolchain to learn/pin; recipe quirks for gRPC static.
- **Effort:** Low-Medium (comparable to A).

### Option D — musl static (true single binary, T2)

Build the CLI and the whole stack against a musl toolchain
(`x86_64-linux-musl`, e.g. inside Alpine) for a `-static` binary with no loader.

- **Pros:** the only route to T2 — a genuinely dependency-free single file that
  runs on any x86-64 Linux kernel; musl static DNS has none of glibc's NSS
  `dlopen` problems.
- **Cons:** must build *every* dependency against musl (no distro shortcuts);
  some libs need small patches; CI must run in an Alpine/musl container.
- **Effort:** High. Best pursued as a follow-on after A/C prove out T1.

### Option E — Build-in-container, ship the binary

Orthogonal to A–D: run the chosen build inside a pinned container
(`fedora:43` for T1, `alpine:edge` for T2) so the toolchain is reproducible and
the host stays clean.

- **Recommendation:** adopt regardless of A/C/D — it is the delivery mechanism,
  not a competing strategy.

---

## 7. Recommendation

1. **T1 via Option A (vcpkg static triplet), built in a container (Option E).**
   Best effort-to-result ratio: vcpkg owns the dependency graph; container
   gives reproducibility; glibc stays dynamic so DNS/NSS is safe.
2. **First, shrink the problem:** confirm and drop the unused `libpq` link (§3),
   and audit whether the CLI needs OpenSSL directly or only transitively via
   gRPC TLS (it needs TLS, so OpenSSL stays).
3. **T2 (Option D, musl)** as a later, optional track if a literal zero-dep
   single file is required for distribution.

## 8. Implementation plan

**Phase 0 — Shrink & measure (½ day)**
- `nm -DC fileengine_cli_static | grep ' U '` to enumerate the real external
  symbol set. Confirm no `PQ*` (drop libpq) and catalogue gRPC/protobuf/OpenSSL
  symbol groups.
- Record a baseline `ldd` and binary size.

**Phase 1 — vcpkg static stack (2–3 days)**
- Add `vcpkg.json` manifest pinning `grpc`, `protobuf` (+ a baseline commit).
- Add a `cmake/static-cli.cmake` that, when `BUILD_STATIC_CLI=ON` *and* a
  vcpkg/toolchain is present, `find_package`s the static targets and links the
  CLI against them instead of the system shared libs.
- Generate `fileservice.pb.*` with the vcpkg `protoc` (not the system one).
- Link with `--start-group` for the static archive set.

**Phase 2 — Verify T1 (1 day)**
- `ldd` shows only glibc + loader (+ `libm`/`libpthread`/`libdl`).
- `ldd -r` reports no unresolved symbols.
- Run `tests/cli_full_integration.sh` against a live server using the static
  binary (set `CLI=` to its path) — must stay 74/74.
- Multi-distro smoke test (see §10).

**Phase 3 — Packaging (½ day)**
- Point the `fileengine-cli-static` RPM subpackage at the T1 binary. Its
  auto-generated `Requires` should collapse to glibc only.
- Also publish a plain `.tar.gz`/`.xz` with the single binary + LICENSE notices
  for non-RPM distros.

**Phase 4 (optional) — T2 musl track (3–5 days)**
- Alpine build container; rebuild the stack + CLI `-static`; verify
  `file` reports "statically linked" and the binary runs in a `scratch`
  container.

## 9. Size management

A fully static gRPC binary is large (tens of MB), dominated by `libicudata`
(~30 MB) and the gRPC/abseil object set.

- Investigate building protobuf/abseil **without ICU** (the CLI does no
  Unicode-collation work) — potentially removes the single biggest contributor.
- `strip --strip-unneeded` (T0 binary drops ~13 MB → ~2 MB).
- `-ffunction-sections -fdata-sections -Wl,--gc-sections` to drop unreferenced
  code at link time.
- Optionally compress with `upx` for distribution (decompresses on exec).

Document the expected final size in the deliverable; "fat but self-contained"
is the explicit tradeoff for the client-only package.

## 10. Verification matrix

| Check | Tool | Pass condition |
|---|---|---|
| No project/libstdc++ deps | `ldd` | only glibc + loader (T1) |
| No unresolved symbols | `ldd -r` | empty |
| Functional parity | `CLI=… tests/cli_full_integration.sh` | 74/74 |
| Old-glibc target | run on the **oldest** supported distro | runs (glibc is forward-compatible, so build on the oldest target glibc) |
| Clean-room | run in a `scratch`/minimal container | runs (T2) or runs with only glibc present (T1) |

Build on the **oldest** glibc we intend to support: glibc symbol versioning is
backward- but not forward-compatible, so a binary built against new glibc fails
on older hosts. A CentOS-Stream/UBI base in the build container pins this.

## 11. Risks

| Risk | Likelihood | Mitigation |
|---|---|---|
| Abseil/gRPC/protobuf version skew at link | High if hand-rolled | Use vcpkg/Conan to own the graph (Options A/C) |
| OpenSSL still `dlopen`s providers | Medium | `no-dso no-engine`; verify with `ltrace`/`strace` on first run |
| glibc-static NSS/DNS breakage | High if T2-on-glibc attempted | Keep glibc dynamic for T1; use musl for T2 |
| Binary too large to be practical | Medium | §9 size management; ICU removal |
| `protoc` mismatch | Medium | Generate protos with the same toolchain protobuf |
| Build time hurts CI | Medium | Cache vcpkg/Conan binaries; build in a reusable container layer |

## 12. Open questions / decisions needed

1. **Target floor:** is T1 (glibc dynamic) acceptable, or is a literal
   single-file (T2/musl) a hard requirement for distribution?
2. **Toolchain:** vcpkg vs Conan — any existing org preference?
3. **Oldest supported glibc / distro** to fix the build base image.
4. **Drop libpq from the CLI link** outright (pending the Phase 0 symbol audit)?
5. **Size ceiling** — is ~tens-of-MB acceptable, or must we strip ICU / UPX?
6. **Distribution channel** beyond RPM (raw tarball, container image)?
